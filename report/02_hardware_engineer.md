The single source of truth for which DAC drives what is `MUXInitialize.py` in each project. The
two active projects share an identical `BaseConfig` because the same RFSoC board is wired the same
way in both:

- `D:/Agentic_QSim_Measurement/WorkingProjects/triangle_lattice_quench/MUXInitialize.py:29-49`
- `D:/Agentic_QSim_Measurement/WorkingProjects/Triangle_Lattice_tProcV2/MUXInitialize.py:29-49`

```python
BaseConfig = {
    "res_ch": 8,                # readout DAC channel (drives res_drive into the cavity input)
    "qubit_ch": 9,              # qubit drive DAC channel
    "ro_chs": [0],              # ADC channel(s); overwritten when MUX has multiple readouts
    "fast_flux_chs": [0..7],    # 8 FF DAC channels, one per qubit
    "res_nqz": 1,
    "qubit_nqz": 2,
    "mixer_freq": -1750,        # MHz, readout-DAC fine NCO mixer
    "qubit_mixer_freq": 4000,   # MHz, qubit-DAC fine NCO mixer
    "res_LO":  9000,            # MHz, external LO mixed up at room temp
    "qubit_LO": 0,              # MHz; 0 means no external mixing on the qubit line
    "relax_delay": 200,         # us
    "res_phase": 0,
    "res_length": 10,           # us, length of the cavity readout pulse
}
```

### 1.1 What each field physically means

**`res_ch = 8`** — RFSoC DAC channel #8 is the *readout* generator. It is declared as a MUX
generator (multiple tones from a single DAC, see Section 4) and feeds the readout cavity input
chain (room-temperature mixer, IR/cryo attenuators, then the feedline). All resonator probe tones
ride this single output.

**`qubit_ch = 9`** — DAC channel #9 is the *qubit drive* generator. It is a single-tone
arb/const generator producing the qubit XY drive, mixed up by the qubit LO when one is used.

**`ro_chs = [0]`** — ADC channel #0 is the readout digitizer. The list grows when more readouts
are used in MUX (`UPDATE_CONFIG.py:42` rewrites it: `config['ro_chs'] = [i for i in
range(len(Qubit_Readout))]` — i.e. channels `[0]`, `[0,1]`, `[0,1,2]` etc, one ADC per multiplexed
readout slot).

**`fast_flux_chs = [0,1,2,3,4,5,6,7]`** — DAC channels 0–7 are the eight fast-flux outputs, one
per qubit on the 8Q triangle lattice device. These drive the on-chip flux lines that bias the
SQUID loops and produce the ramps and sudden flux pulses that implement the quench Hamiltonian.
See Section 2 for full details.

**`res_nqz = 1`** — Readout DAC operates in **Nyquist zone 1**. For the QICK RFSoC board family,
this means the DAC output filter is set so the *first* image (DC up to fs/2) is the usable band.
Concretely: with the readout chain set up around 7–8 GHz, the IF generated by the DAC + NCO is
mixed up with `res_LO = 9 GHz`, and the DAC itself produces tones in nqz=1 (≈ 0–4.9 GHz), so
`mixer_freq + res_freqs` lives below the DAC fs/2.

**`qubit_nqz = 2`** — Qubit DAC operates in **Nyquist zone 2**. The qubit lives around 3.5–4.5
GHz (resonance ≈ 3800 MHz, see e.g. `Qubit_Parameters.py:64`), which is *above* fs/2 of the DAC
in zone 1. nqz=2 tells QICK to use the upper image of the DAC (between fs/2 and fs); the inverse
sin(x)/x roll-off and image filter are configured accordingly. The qubit drive is therefore
generated *directly* at RF (no qubit_LO needed; `qubit_LO = 0`).

**`mixer_freq = -1750` MHz** — fine-grained NCO/mixer frequency programmed into the readout DAC's
internal mixer. With `res_LO = 9000` MHz, this places the DAC tone center at
`9000 + (-1750) = 7250 MHz`, which is in the middle of the cavity band (resonator dressed
frequencies are around 7.1–7.6 GHz, see `Qubit_Parameters.py:93-108`). Individual MUX readout
tones `res_freqs` are then offsets relative to this center.

**`qubit_mixer_freq = 4000` MHz** — fine NCO of the qubit DAC. Because `qubit_LO = 0`, the actual
qubit RF frequency on the line is just `qubit_mixer_freq + qubit_freq_offset`. The header
explicitly notes:

> `# range=1720.320 MHz, so allowed qubit freqs will be qubit_mixer_freq +- 860 MHz`
> `# e.g. 3140 MHz to 4860 MHz if mixer at 4000 MHz`
> (`MUXInitialize.py:39-41`)

So if you need to drive a qubit far from 4 GHz, you must move the mixer center first — sweep
ranges that exceed ±860 MHz around the mixer will silently fold, alias, or simply not generate.

**`res_LO = 9000` MHz** — external local oscillator mixed with the DAC IF on the readout
**transmit** side at room temperature, and with the ADC IF on the **receive** side (typical
heterodyne layout). The reported readout frequency in display code is always
`res_freqs[i] + res_LO` (see `mTransmissionFFMUX.py:141`,
`mSpecSliceFFMUX.py:72,98`).

**`qubit_LO = 0`** — no external LO on the qubit drive; the DAC running in nqz=2 directly produces
the RF. If the team ever needs to push the qubit band well outside ±860 MHz around 4 GHz, an
external LO would have to be added and `qubit_LO` and `qubit_mixer_freq` reorganized.

### 1.2 Nyquist zones, in plain language

The QICK RFSoC DAC samples at fs ≈ 9.83 GS/s (zone boundary fs/2 ≈ 4.92 GHz on this board).

- **nqz=1**: usable band ≈ 0 → fs/2. DAC output filter passes the baseband image. Use this when
  the signal you want is *below* fs/2 (here: the readout IF, then mixed up to ~7 GHz with an
  external 9 GHz LO).
- **nqz=2**: usable band ≈ fs/2 → fs. DAC outputs a tone whose fundamental is below fs/2 but
  whose first alias (= the desired RF) is above fs/2; the output filter is configured to pass
  that band. Use this for direct synthesis above fs/2 (here: the qubit drive at ~3.5–4.5 GHz, no
  external LO).

Picking the wrong nqz produces a spectrally inverted, attenuated, or completely filtered-out
tone. It does **not** physically damage anything, but the experiment will look broken in
non-obvious ways (e.g. a Rabi amplitude that scales the wrong way, or no spectroscopy peak).

## 2. Fast-flux (FF) subsystem

The FF subsystem produces eight independent flux waveforms that drive the per-qubit SQUID flux
lines on the chip. These pulses implement: (a) the readout flux point, (b) the experiment flux
point (where qubit frequencies sit during evolution), (c) the pi-pulse flux point (where qubit
spectroscopy/drive is calibrated), (d) the beam-splitter flux point, and (e) any adiabatic ramp
that connects the above.

### 2.1 `FF_Qubits` dictionary

Defined in both projects' `MUXInitialize.py`:

- `WorkingProjects/triangle_lattice_quench/MUXInitialize.py:52-61`
- `WorkingProjects/Triangle_Lattice_tProcV2/MUXInitialize.py:52-61`

```python
FF_Qubits = {
    str(1): {'channel': 0, 'delay_time': 0.000},
    str(2): {'channel': 1, 'delay_time': 0.000},
    str(3): {'channel': 2, 'delay_time': 0.000},
    str(4): {'channel': 3, 'delay_time': 0.000},
    str(5): {'channel': 4, 'delay_time': 0.000},
    str(6): {'channel': 5, 'delay_time': 0.000},
    str(7): {'channel': 6, 'delay_time': 0.000},
    str(8): {'channel': 7, 'delay_time': 0.000},
}
```

Keys are **qubit indices as strings, 1-indexed** ("1" .. "8") — *not* 0-indexed. This is a frequent
source of off-by-one bugs: indexing with `Q-1` is correct only if the dict key is `str(Q)`.

Each entry has:
- `channel`: which RFSoC DAC drives that qubit's flux line. Currently a clean Q1→ch0, Q2→ch1, ...
  Q8→ch7 mapping (i.e. just `Q_idx - 1`). Keep this in sync with the cryostat wiring map.
- `delay_time`: per-qubit fixed time offset, in *us*, that gets added to every FF pulse on that
  channel. It is consumed in `FF_utils.FFDefinitions` at
  `WorkingProjects/Triangle_Lattice_tProcV2/Helpers/FF_utils.py:193,203-205`:
  ```python
  FFDelays = np.array([instance.cfg["FF_Qubits"][q]["delay_time"] for q in instance.FFQubits])
  instance.gen_t0[instance.FFChannels] = [instance.us2cycles(d) for d in FFDelays]
  ```
  Use it to compensate for cable-length mismatches between FF lines (so all qubits see the
  flux step at the same instant). All eight are currently zero — the line lengths are matched, or
  delays are absorbed into the waveform itself ("Lets do this within the waveform from now on" —
  `MUXInitialize.py:51`).

### 2.2 FF gain keywords (`Gain_*`) and `ramp_initial_gain`

These keys are added per-qubit into the `FF_Qubits` dict by
`WorkingProjects/triangle_lattice_quench/Run_Experiments/UPDATE_CONFIG.py:5-24`:

```python
for Qubit, FFR, FFE, FFP, FFBS in zip(('1','2',...,'8'), FFReadouts, FFExpt, FFPulse, FF_BS):
    FF_Qubits[Qubit] |= {'Gain_Readout': FFR, 'Gain_Expt': FFE,
                         'Gain_Pulse': FFP, 'Gain_BS': FFBS}
...
FF_Qubits[Qubit]['ramp_initial_gain'] = FFI
```

All five values are integer DAC gains in QICK gain units (full DAC range = `[-32766, +32766]`).
They are pulled out into per-experiment numpy arrays by `FF_utils.FFDefinitions`
(`Helpers/FF_utils.py:177-190`), which also passes them through `FF_Crosstalk_Helper.correct(...)`
to compensate flux crosstalk before sending to the DAC.

| Key | Lifecycle in a sequence | Physical meaning |
|---|---|---|
| `Gain_Pulse` | Active during the qubit pi-pulse / spectroscopy (`mSingleShotProgramFFMUX.py:44`, `mSpecSliceFFMUX.py:56`). | Flux bias at which qubit_freqs are calibrated; this is where the qubit DAC drive is on resonance. |
| `Gain_Readout` | Active during the cavity readout pulse (`mSingleShotProgramFFMUX.py:56`, `mTransmissionFFMUX.py:33`). | Flux bias for *readout*. Often shifted away from the experiment point so that resonator-dispersive shifts give good single-shot fidelity. |
| `Gain_Expt` | Active during the analog evolution / experiment window. | Flux bias that defines the simulated Hamiltonian (qubit detunings & nearest-neighbor exchange J). The "(3800.0, ...)" comments in `Qubit_Parameters.py:19` describe the mapped-to qubit frequencies in MHz at this gain set. |
| `Gain_BS` | Active during a beam-splitter / parametric coupling pulse. | Flux bias that brings two qubits onto resonance (or onto a parametric drive sweet spot) to enact a swap/beamsplitter. Many `_correlations` entries in `ramp_params` (`Qubit_Parameters.py:157-201`) are tuned BS points. |
| `ramp_initial_gain` | First sample of the FF ramp envelope (entry point of `Init_FF`). | Where the adiabatic ramp **starts** before sweeping into `Gain_Expt`. If `Init_FF` is `None` in the active state, the code falls back to `Gain_Pulse` (`UPDATE_CONFIG.py:21-24`). |

The corresponding arrays as they appear inside a QICK program (constructed in `FFDefinitions`):
- `instance.FFReadouts` = corrected readout gains
- `instance.FFExpts` = corrected experiment gains
- `instance.FFPulse` = corrected pi-pulse gains
- `instance.FFBS` = corrected beam-splitter gains

These are the actual ndarrays handed to `FFPulses(...)` to play a constant flux step
(see `FF_utils.FFPulses` at `Helpers/FF_utils.py:72-103`, gain is encoded as
`gain / 32766` to fit the QICK `[-1, 1]` API).

### 2.3 Step pulses and ramps

Two execution helpers in `FF_utils.py`:

- **`FFPulses(instance, list_of_gains, length_us, ...)`** (line 72): emits a *constant* flux pulse
  of length `length_us` at the listed gains across all 8 channels. Used for the readout box-pulse,
  pi-pulse plateau, and BS plateau.
- **`FFPulses_direct(instance, list_of_gains, length_dt, previous_gains, IQPulseArray, ...)`**
  (line 9): emits an arbitrary I-data envelope per channel, sampled at 16× the fabric clock
  (1/16 clock cycle = ~4.65/16 ns). Used for ramps. Pulse memory is padded so length is
  commensurate with 16 samples (line 43-50) and so total length is at least 3 cycles (line 47-50)
  — a hardware constraint of the QICK arb generator.

`FFEnvelope_Helpers.py` builds the actual ramp arrays:
- `StepPulseArrays(cfg, initial_key, final_key)` — `Compensated_Pulse` step (line 24-30)
- `CubicRampArrays(...)` — cubic adiabatic ramp (line 32-42)
- `LinearRampArrays(...)` — linear ramp (line 73-79)
- `ThreeExpRampArrays(...)` — calibrated three-exponential ramp accounting for flux line LPF (line
  81-87)
- `CompensatedRampArrays(...)` — ramp with a compensated initial jump (line 44-71)

`RampHelpers.py` provides the underlying shape generators:
- `generate_linear_ramp` (line 8)
- `generate_cubic_ramp` (line 32) — interpolates between `initial_gain` and `final_gain` over
  `ramp_duration` clock cycles (4.65/16 ns each); `reverse=False` flattens out near `final_gain`,
  `reverse=True` flattens out near `initial_gain`.
- `generate_exp_ramp` (line 73) — exponential ramp.
- `generate_three_exp_ramp` (line 109) — uses fitted coefficients
  `[20.009, -0.0671, 4.679, -544.4, -0.117]` that compensate for the measured impulse response of
  the flux line (the on-chip RC + cable LPF). The hardcoded coefficients in this function are an
  **implicit calibration** — re-run the flux line response calibration if you change cabling,
  bias-tee, or the cryo configuration.

The fundamental time unit for these ramps is **1/16 of a fabric cycle** ≈ **4.65/16 ns ≈ 0.29 ns**
(see the docstring at `RampHelpers.py:14` and `FF_utils.py:14-19`). This is the resolution at
which an FF envelope sample is emitted.

### 2.4 `Additional_Delays`

`MUXInitialize.py:63-67`:
```python
Additional_Delays = {
    str(1): {'channel': 4, 'delay_time': 0}
}
BaseConfig["Additional_Delays"] = Additional_Delays
```

This is a *secondary* per-channel delay table, applied on top of `gen_t0` in
`FFDefinitions` (`Helpers/FF_utils.py:199-205`). Unlike `FF_Qubits[*]['delay_time']` which is
indexed by qubit, `Additional_Delays` is indexed by an arbitrary key (here just "1") and contains
a `channel` and a `delay_time`. Use it to delay any *non-FF* generator (e.g. the qubit drive
channel, the readout channel, or even an FF channel to override its `FF_Qubits` delay).

In the current config it's set to channel 4 with `delay_time = 0`, i.e. effectively a no-op
placeholder. It exists as a hook: if you find that the qubit pulse arrives ≈10 ns ahead of the FF
plateau (or vice versa), this is the place to tune it without touching `FF_Qubits`.

### 2.5 FF crosstalk correction

`Helpers/FF_Crosstalk_Helper.py:8-26` loads an 8×8 crosstalk matrix from
`Z:/QSimMeasurements/Measurements/8QV1_Triangle_Lattice/qubit_parameters/FF_crosstalk_3.csv`
at import time and inverts it. Every gain vector that flows through `FFDefinitions` or
`get_gains` is multiplied by `FF_CORRECTION = inv(FF_CROSSTALK)` before reaching the DAC. If the Z:
drive isn't mounted, you'll see "No FF crosstalk correction applied." in the console — the code
will continue running but the requested flux points will be off by the crosstalk amount.
**Re-measure the matrix when wiring or chip changes**.

## 3. External instruments (room-temperature)

The PythonDrivers folders contain the drivers for the room-temperature instruments that sit
between the RFSoC outputs and the cryostat inputs (and the DC bias coil drivers).

### 3.1 Vaunix LDA — variable RF attenuators

- Driver: `D:/Agentic_QSim_Measurement/MasterProject/Client_modules/PythonDrivers/control_atten.py`
- DLL: `D:/Agentic_QSim_Measurement/MasterProject/Client_modules/PythonDrivers/VNX_atten64.dll`

`setatten(attenu, serial, printv)` (`control_atten.py:8-86`):
- enumerates up to 5 connected LDA devices via `fnLDA_GetDevInfo` (line 25),
- looks up the unit by USB serial number (line 30-33),
- sets attenuation in 0.05-dB steps via `fnLDA_SetAttenuationHR(serial, atten/0.05)` (line 50-60).

Physically, the LDA is a USB-controlled step attenuator. In this rig there are typically separate
LDA units in the **readout drive line**, **qubit drive line**, and possibly the **return/RF
amplifier path**, each addressed by its serial number (line 89: `serial = 27786` is the example
default in `__main__`). These set the **room-temperature attenuation** on the way down to the
fridge — the chip-side power is `DAC gain × room-temp atten + cryo atten`. The cryo attenuators
themselves (typically 20 dB at 4K, 20 dB at still, 20 dB at MXC) are **fixed** and not under
software control; tune the LDA to set chip-side power.

The LDA sees a hardware Nyquist-equivalent constraint: it is rated for some band (e.g. DC–18 GHz
for an LDA-602Q); using it outside its band silently degrades. Verify the LDA model from its
sticker matches the band you're operating in.

### 3.2 Yokogawa GS200 — DC source

- Driver: `D:/Agentic_QSim_Measurement/MasterProject/Client_modules/PythonDrivers/YOKOGS200.py`

## 4. MUX readout architecture

The readout side uses **frequency multiplexing**: a single readout DAC (`res_ch = 8`) generates
multiple resonator probe tones simultaneously, one per resonator being read, and a single ADC
(`ro_chs[0] = 0`, possibly more) digitizes the combined return signal. QICK then digitally
demodulates each tone independently inside the FPGA.

### 4.1 Code wiring

`mTransmissionFFMUX.py:17-23`:
```python
self.declare_gen(ch=cfg['res_ch'], ro_ch=cfg['ro_chs'][0], nqz=cfg["res_nqz"],
                 mixer_freq=cfg["mixer_freq"],
                 mux_freqs=cfg['res_freqs'],
                 mux_gains=cfg['res_gains'],
                 mux_phases=[0]*len(cfg['res_freqs']))
for ch, f in zip(cfg['ro_chs'], cfg['res_freqs']):
    self.declare_readout(ch=ch, length=cfg['readout_length'], freq=f, phase=0,
                         gen_ch=cfg['res_ch'])
```

Same pattern in `mSingleShotProgramFFMUX.py:18-25` and `mSpecSliceFFMUX.py:18-25`.

The pulse itself is a constant-amplitude box of length `cfg["res_length"]` with `mask=cfg["ro_chs"]`
(`mTransmissionFFMUX.py:25-28`). The `mask` is the *MUX mask*: a bitmask that tells the DAC which
of the (up to N) MUX tones to actually emit on this pulse — passing the list of ADC channel
indices means "emit tone *i* of the MUX bank if the ADC at index *i* is being triggered now".

### 4.2 The lists — one entry per readout channel

| Field | Type | Meaning |
|---|---|---|
| `res_freqs` | list[MHz] | Resonator probe frequency *offsets* from `mixer_freq` (so the actual RF tone is `res_LO + mixer_freq + res_freqs[i]`). One entry per readout slot. Built in `UPDATE_CONFIG.py:30` as `Qubit_Parameters[Q]['Readout']['Frequency'] - res_LO`. |
| `res_gains` | list[fraction] | DAC amplitude fraction for each MUX tone, ∈ [0, 1] (at full output) or [-1, 1] for signed. Built in `UPDATE_CONFIG.py:29` as `Gain / 32766. * len(Qubit_Readout)` — the multiplication by `len(Qubit_Readout)` *normalizes* per-tone gain so that the summed waveform (which is what the DAC actually outputs) doesn't saturate. **Watch for this**: as you grow N readout channels, per-tone amplitude should scale appropriately, but the calibration `Gain` values in `Qubit_Parameters.py:93-108` are stored *per single-tone* — the multiplication is the conversion. |
| `readout_lengths` | list[us] | ADC integration window per readout channel, in microseconds. Set per qubit in `Qubit_Parameters` as `'Readout_Time': 3` (us) — `UPDATE_CONFIG.py:31`. Different qubits can have different optimal integration times. |
| `adc_trig_delays` | list[us] | Per-channel trigger offset between the start of the readout pulse and the start of ADC integration. Set as `'ADC_Offset': 1` (us) in `Qubit_Parameters` — `UPDATE_CONFIG.py:32`. Compensates for the cable round-trip delay and the resonator ring-up time of each individual cavity. |

The body code triggers each ADC at its own delay and emits the multiplexed pulse:

`mTransmissionFFMUX.py:32-43`:
```python
def _body(self, cfg):
    self.FFPulses(self.FFReadouts, self.cfg["res_length"])
    self.delay(0.1)
    for ro_ch, adc_trig_delay in zip(self.cfg["ro_chs"], self.cfg["adc_trig_delays"]):
        self.trigger(ros=[ro_ch], pins=[0], t=adc_trig_delay)
    self.pulse(self.cfg["res_ch"], name='res_pulse')
    self.wait_auto()
    self.delay_auto(10)
    self.FFPulses(-1 * self.FFReadouts, self.cfg["res_length"])
```

The trailing `FFPulses(-1 * self.FFReadouts, ...)` is a **DC-balance** kick: invert the readout
flux step for the same length as the pulse to keep the average flux zero on every trial. Without
it, repeated readout pulses would accumulate a DC drift in the bias-T inductor.

### 4.3 The `Qubit_Readout_List` and `ro_chs` length

`UPDATE_CONFIG.py:42`:
```python
config['ro_chs'] = [i for i in range(len(Qubit_Readout))]
```

So the number of ADC channels actually used is `len(Qubit_Readout)`. Each of `res_freqs`,
`res_gains`, `readout_lengths`, `adc_trig_delays` must have length equal to `len(Qubit_Readout)`,
and the indexing convention is: position `i` in each of these lists corresponds to **ADC channel
`i`**, which is reading qubit `Qubit_Readout[i]`. `Qubit_Readout_List` is stored back into the
config (line 41) so downstream histogram analysis can label IQ clouds by qubit number.

### 4.4 MUX readout limits

- **Number of tones** is bounded by the bitstream. The 8Q triangle-lattice configurations read up
  to 8 qubits at a time on a single MUX generator. Verify `soccfg` at the top of a run if you
  intend to read more than 8.
- **Tone spacing** must be wider than `1 / readout_length` to avoid spectral leakage between
  channels in the digital demodulator. With `Readout_Time = 3 us`, that's ~330 kHz minimum
  spacing; the actual cavity tones in this device are MHz apart, so this is comfortable.
- **Total summed amplitude** at the DAC must stay within ±1 (in DAC fractional units). The
  `* len(Qubit_Readout)` in the gain build is what makes per-tone amplitudes scale down as the
  number of tones grows; the *individual* `Gain` value in `Qubit_Parameters.py` should always
  be the per-tone optimum. If you ever see the readout I/Q clip flat at large gains, that's the
  summed-amplitude limit being hit — the DAC envelope clips before the digital filter.

### 5.2 DAC range constraints

- **Qubit DAC mixer range**: ±860 MHz around `qubit_mixer_freq`. With mixer at 4000 MHz, allowed
  qubit drives are 3140–4860 MHz (`MUXInitialize.py:39-41`). If the chip's qubit happens to land
  outside this, change `qubit_mixer_freq` and re-validate Nyquist (the new center must still be
  in nqz=2 of the DAC, i.e. between fs/2 ≈ 4.92 GHz and fs ≈ 9.83 GHz on the lower-nqz side).
- **Readout DAC mixer range**: same ±860 MHz around `mixer_freq`, but in nqz=1. With
  `mixer_freq = -1750` and `res_LO = 9000`, the center is at `9000 - 1750 = 7250 MHz`; readouts
  in `Qubit_Parameters.py:93-108` range from 7121.5 to 7568.4 MHz, all within ±860 MHz — fine.
- **FF gain range**: `[-32766, +32766]` (signed 16-bit), enforced by clipping in
  `FF_utils.FFPulses_direct` at lines 35-39:
  ```python
  IQPulse[IQPulse < -gencfg['maxv']] = -gencfg['maxv']
  IQPulse[IQPulse >  gencfg['maxv']] =  gencfg['maxv']
  ```
  After crosstalk correction (`FF_Crosstalk_Helper.correct`, line 22-25) the same clip happens.
  Going close to ±32766 means physical clipping in DAC volts → distorted flux waveform → wrong
  experiment Hamiltonian. Calibrations should aim for headroom (the canonical Expt_FF in
  `Qubit_Parameters.py:20` lives in the ±10000 range, well clear of saturation).
- **MUX readout total amplitude**: see Section 4.4. Sum of `res_gains` should stay below 1 in
  fractional DAC units to avoid digital saturation in the multiplexer.

### 5.4 Boot-up sanity checklist (before any RF on the chip)

1. **Z: drive mounted?** Test by `ls Z:\QSimMeasurements\Measurements\8QV1_Triangle_Lattice\`. If
   not, FF crosstalk correction silently disables (`FF_Crosstalk_Helper.py:13-16`).
2. **RFSoC reachable?** `ping 192.168.1.114` and check the Pyro4 nameserver is up on port 8888.
3. **SPI rack on COM3, powered?** Open a Python shell and import `SPIRack`; if Windows reassigned
   the port (e.g. after a USB unplug) every flux-bias script breaks at line 5.
4. **D5a span set to bipolar?** The script auto-sets `range_4V_bi` on import
   (`QbloxVoltageSet_8QTriangleLattice.py:9-17`), but only when run; if you skipped it after a
   power cycle, `set_voltage_ramp` will refuse on `range_4V_uni`
   (`SPIRackvoltage.py:41-42`).
5. **Vaunix LDA serial numbers correct?** Each script calling `setatten` references a fixed
   serial; if a unit was swapped, attenuation will not change and chip-side power can be wrong by
   tens of dB.
6. **FF crosstalk matrix age?** `FF_crosstalk_3.csv` is loaded at import (line 8). Re-measure if
   wiring changed since the last calibration (re-bonded chip, swapped fridge cable, etc.).
7. **`mixer_freq` and `qubit_mixer_freq` consistent with the qubits at the current
   `Gain_Pulse`?** A qubit detuned by 1 GHz from `qubit_mixer_freq + 0` will simply not respond
   — verify spec freqs are within ±860 MHz of the mixer center *for the FF point you're driving
   at*.

### 5.5 Hardware prerequisites a calibration depends on

(Hardware-only view — what must be physically true before software-level calibration runs.)

| Calibration | Hardware prereqs |
|---|---|
| Resonator transmission (Section 4 scripts) | RFSoC + LO synced, LDA in readout drive line known, ADC/DAC in nqz=1, MUX gen declared with `mixer_freq` set so all `res_freqs` lie within ±860 MHz of mixer. FF DC bias set so cavities sit at their dispersive-shifted parking frequencies. |
| Single-shot readout fidelity (`mSingleShotProgramFFMUX`) | Above, plus: qubit DAC nqz=2 with mixer at the qubit band, FF `Gain_Pulse` and `Gain_Readout` calibrated so the qubit responds at one and the cavity reads contrast at the other. Any TWPA/HEMT in the return path biased correctly (TWPA pump on if used). |
| Qubit spectroscopy (`mSpecSliceFFMUX`) | All of the above, plus FF `Gain_Pulse` set to the experiment-relevant flux point. Sweep range should stay within ±860 MHz of `qubit_mixer_freq`; otherwise increase the sweep in multiple slices, each with a different mixer center. |
| FF ramp / quench experiment | Stable D5a DC bias on all 8 qubits + 6 couplers; FF crosstalk matrix valid; `Gain_Expt`, `Gain_BS`, `ramp_initial_gain` defined for every qubit; `Additional_Delays` set so the qubit drive arrives in sync with the FF plateau. |
